# 第2章 AI 的发展历史

**Knowledge Tree:** Part I 世界观：AI 为什么会发展成今天这样
**Status:** Draft

**Roadmap Intent:** 从规则系统到机器学习、深度学习、Transformer、LLM、Agent，理解每次范式变化背后的瓶颈。

## 本章要回答的问题

AI 的历史不应该被理解成一串年份、人物和技术名词。

对 AI System 的学习来说，更重要的问题是：为什么 AI 会从规则系统走向机器学习，从机器学习走向深度学习，从深度学习走向 Transformer 和 LLM，又为什么 LLM 之后会继续走向 RAG、Tool Calling、Memory 和 Agent？

这些变化不是简单的技术潮流。每一次范式变化，背后都有一个更深的系统瓶颈：

- 规则系统解决了显式知识表达的问题，却遇到知识覆盖和维护成本的瓶颈。
- 机器学习把规则从手写变成从数据中学习，却遇到特征工程和分布泛化的瓶颈。
- 深度学习把特征也交给模型学习，却把瓶颈推向数据规模、算力、优化和硬件效率。
- Transformer 改变了序列建模方式，却把瓶颈推向注意力计算、显存、上下文长度和训练规模。
- LLM 通过规模化预训练形成通用能力，却把瓶颈推向对齐、推理成本、运行时上下文和可靠性。
- Agent 把模型连接到工具和业务流程，却把瓶颈推向权限、状态、记忆、动作边界、观测和治理。

这条线索是一种用于理解 AI System 的分析框架，不是对全部 AI 历史的单因果解释。规则、统计学习、深度学习和符号方法并没有按年份依次消失；它们至今仍在不同约束下共存。所谓“范式变化”，更准确地说，是某类方法开始经济地承担过去难以承担的问题，而不是旧方法被彻底淘汰。

所以，本章不是要背 AI 发展史，而是要建立一个判断框架：**AI 的历史，是瓶颈不断迁移的历史；AI System 的历史，是这些瓶颈逐步进入工程系统边界的历史。**

## 如果从零开始设计智能系统

先不要急着看现代答案。假设我们完全不知道今天的 LLM，只是想造一个能处理复杂任务的系统。

最自然的第一步，是把人的知识写成规则。

如果输入满足某些条件，就执行某个动作；如果出现某类事实，就推出某个结论。对软件工程师来说，这很熟悉。传统后端系统、风控系统、工作流引擎、告警规则、权限系统，本质上都大量依赖这种思想。

规则系统有明显优点。它可解释、可审计、行为边界清晰，尤其适合约束明确的领域。你能知道系统为什么得出某个结论，也能通过修改规则来修正行为。

但问题很快出现：真实世界太复杂，规则写不完。

当输入空间很小、业务规则稳定时，手写规则是有效的；当输入变成自然语言、图像、语音、用户行为、复杂环境状态时，规则数量会爆炸。更麻烦的是，规则之间会互相冲突，规则维护需要领域专家长期投入，系统遇到边界情况时往往非常脆弱。

于是第一个根本问题出现了：

```text
能不能不要手写每一条规则，
而是让系统从样本中自己归纳规律？
```

这就是机器学习的核心动机。

## 从手写规则到从数据中学习

机器学习改变了系统能力的来源。

在规则系统中，能力来自人写进程序里的知识。在机器学习中，能力来自数据、特征、模型假设和优化过程。系统不再直接写出“如果 A 则 B”的完整规则，而是给定一批样本，让模型学习一个从输入到输出的函数。

可以把这个变化抽象成：

```text
Rule System:
human knowledge -> explicit rules -> output

Machine Learning:
data + labels + model class + optimization -> learned function -> output
```

这看起来只是技术实现变化，但工程含义非常深。

第一，系统的核心资产开始从代码扩展到数据。以前修复系统行为主要是改代码；现在模型效果不好，可能不是代码错了，而是数据分布、标签质量、特征设计、训练集覆盖、评估方式出了问题。

第二，系统行为从确定性逻辑变成统计泛化。模型并不是记住所有样本，而是在有限样本上学习某种规律，并希望这种规律能泛化到未见过的数据。这就带来一个长期问题：训练数据和真实世界是否同分布？评估集是否代表线上流量？模型在边界输入上会怎样？

第三，工程流程开始出现 MLOps 的雏形。数据版本、训练任务、模型评估、模型上线、线上反馈，逐渐成为系统的一部分。一个模型不是编译产物，而是数据和训练过程共同产生的资产。

但传统机器学习仍然有一个很强的前提：人要把原始世界转换成合适的特征。

比如在搜索、推荐、广告、风控、分类任务中，特征工程曾经是核心工作。模型本身可能是线性模型、树模型、SVM 或其他统计模型，但模型能学到什么，很大程度取决于人如何设计输入表示。

这带来第二个根本瓶颈：

```text
如果特征决定上限，
能不能让模型自己学习表示？
```

## 深度学习：从特征工程到表示学习

深度学习的关键变化，不只是“神经网络层数更多”，而是它把一部分特征工程变成了模型内部的表示学习。

在传统机器学习中，人通常要先定义特征。图像里可能是边缘、纹理、形状；文本里可能是 n-gram、词频、词性；推荐系统里可能是统计特征、交叉特征、行为窗口。模型在这些人为构造的特征上学习。

深度学习试图让模型从原始或更接近原始的数据中学习多层表示。低层表示捕捉局部模式，高层表示组合成更抽象的结构。这个思想在图像、语音、文本等领域都产生了深远影响。

从第一性原理看，深度学习做了一个重要替换：

```text
hand-crafted features + shallow model
        ↓
learned representations + deep model
```

这不是免费午餐。表示学习减少了手工特征工程，但把系统瓶颈推向了新的地方。

首先是数据。模型参数更多，表达能力更强，也更容易过拟合或学习到数据中的偏差。数据规模、数据质量、数据分布和数据清洗开始决定模型上限。

其次是算力。深度网络需要大量矩阵计算，GPU 逐渐成为核心基础设施。训练不再只是算法问题，而是硬件利用率、显存、吞吐、分布式通信和作业调度问题。

再次是优化。深层网络的训练依赖反向传播、初始化、归一化、优化器、学习率策略、正则化等机制。模型能不能训起来，常常取决于一组工程和数学细节的共同配合。

最后是可解释性。规则系统的逻辑可追踪，传统模型的某些特征影响可以分析，而深度网络学习到的是高维表示。能力增强的同时，系统更难解释和治理。

因此，深度学习并没有消除系统问题。它只是把系统问题从“如何写规则、如何做特征”，迁移到了“如何组织数据、算力、训练和评估”。

对 AI System 来说，这是一个重要转折点：**模型能力开始强依赖计算基础设施。**

## 序列建模的旧瓶颈

在 LLM 之前，深度学习已经在图像、语音和 NLP 中取得了重要进展。但自然语言有一个特殊困难：语言不是孤立特征集合，而是序列。

一个词的含义依赖上下文。句子前面的内容会影响后面的解释，后面的内容也可能反过来改变我们对前文的理解。更长的文档、对话和代码，还涉及跨句、跨段、跨文件的依赖。

早期序列模型有多种路线。RNN 按时间步递归处理序列，理论上可以把历史信息传递下去；LSTM、GRU 等结构试图缓解长期依赖问题；CNN 也被用于序列任务，通过局部窗口和层叠扩大感受野；seq2seq 模型把输入序列编码成表示，再生成输出序列。

这些方案都有价值，但也暴露出几个系统性瓶颈。

第一，递归结构天然限制并行度。RNN 必须按顺序处理 token，训练和推理都很难充分利用大规模并行硬件。对小模型、小数据时代这还能接受；当数据和模型规模继续扩大时，硬件效率会成为核心约束。

第二，长程依赖很难稳定传递。即使有门控机制，信息也要经过很多时间步。越长的序列，越难保证远距离信息被有效使用。

第三，固定向量瓶颈限制表达。早期 encoder-decoder 结构如果把整个输入压缩进一个固定表示，长输入的信息会被压缩和丢失。Attention 的早期引入，正是为了让 decoder 在生成时可以回看输入的不同部分。

这时的问题变成：

```text
能不能让序列中的每个位置，
直接根据内容找到它需要关注的其他位置，
同时又适合 GPU 并行计算？
```

Transformer 正是在这个问题上改变了路径。

## Transformer：内容相关的路由机制

Transformer 的关键不是多了几个矩阵公式，而是它改变了序列中信息流动的方式。

在 Self Attention 中，每个 token 不再只依赖固定窗口或递归状态，而是可以根据当前内容，与序列中其他 token 建立关联。它把序列建模从“沿时间逐步传递信息”，改成“在同一层中进行内容相关的信息聚合”。

可以粗略理解为：

```text
RNN:
state_t = f(state_{t-1}, token_t)

Self Attention:
representation_i = weighted_sum(all token representations)
weights depend on content similarity
```

这个变化带来三个重要结果。

第一，依赖路径变短。两个相距很远的 token，可以通过一次 attention 直接交互，而不必把信息沿着许多时间步传过去。

第二，训练并行度更高。序列中的多个位置可以并行计算 attention，这更适合 GPU/TPU 上的大规模矩阵运算。

第三，模型结构更适合扩展。Transformer 层可以重复堆叠，配合大规模数据和算力，逐渐成为预训练语言模型的主流骨架。

但 Transformer 也创造了新的瓶颈。

Attention 的计算和显存开销会随序列长度增长而快速上升。训练时，模型参数、激活、优化器状态和通信会压迫 GPU 集群；推理时，KV Cache 会成为核心运行时状态；长上下文会带来显存、延迟和调度压力。

因此，Transformer 不是让系统变简单了，而是让模型能力进一步变强，同时把基础设施问题推到台前。

后面的 Part II 会详细解释 Attention、Transformer Layer、Decoder Only、KV Cache 等机制；Part IV 会从推理系统角度解释 Prefill、Decode、PagedAttention 和调度为什么会出现。这里先抓住主线：**Transformer 的成功，本质上是模型机制和硬件并行能力之间的一次重要对齐。**

## LLM：从任务模型到通用能力载体

Transformer 本身还不是 LLM。LLM 的关键变化，是把 Transformer 架构、大规模文本数据、大规模算力和自监督目标结合起来，形成一种通用能力生产方式。

最典型的训练目标是预测下一个 token：

```text
given tokens x_1, x_2, ..., x_{t-1}
predict x_t
```

这个目标看起来简单，但它迫使模型从海量文本中学习语言结构、事实关联、代码模式、推理痕迹、对话习惯、写作风格和世界知识的统计关系。模型不是被显式教会每个任务，而是在预训练过程中学习一个通用的条件生成分布。

这带来一个重要范式变化：

```text
Traditional ML:
one task -> one dataset -> one model or one head

LLM:
general pretraining -> prompt/context specifies task at runtime
```

GPT-3 之后，in-context learning 和 few-shot prompting 让人们看到一种新的接口形态：很多任务不再一定通过重新训练模型来定义，而可以通过 prompt、示例和上下文在运行时定义。

这对系统设计的影响很大。

第一，任务边界从训练期移动到运行期。传统模型上线后，输入输出模式相对固定；LLM 的行为强依赖 prompt、system instruction、context、retrieval result、tool result 和 decoding strategy。

第二，模型服务从一次前向计算变成自回归生成过程。生成一个答案需要逐 token decode，前面的输出又成为后续输入。推理系统必须处理流式返回、KV Cache、batching、长尾请求和 token-level latency。

第三，模型从单任务组件变成平台级能力。一个 LLM 可以服务代码生成、问答、摘要、分类、规划、数据分析等多类任务，但每类任务的质量、成本、风险和治理方式不同。

第四，评估变得更难。传统分类任务可以用 accuracy、precision、recall 等指标；LLM 输出开放文本，质量可能包括事实正确性、推理过程、格式稳定性、风格、语气、安全性和业务可接受性。

所以 LLM 的历史意义不只是模型更大，而是它把 AI System 的边界显著扩大了。模型本身仍然重要，但运行时上下文、推理基础设施、评估体系和治理机制开始同样重要。

## 从能力到可用：对齐和后训练为什么重要

预训练让模型获得通用语言能力，但通用能力不等于可用产品。

一个只经过预训练的语言模型，本质上是在续写文本。用户希望它回答问题、遵循指令、拒绝不安全请求、保持格式、承认不确定性、使用工具、解释推理过程，这些都不是“预测下一个 token”目标天然保证的。

这就是 Instruction Tuning、SFT、RLHF、DPO、GRPO 等后训练方法出现的背景。它们试图把基础模型的通用能力，对齐到人类偏好、任务要求和产品约束上。

可以把这个阶段理解为：

```text
Pretraining:
learn broad capability from large-scale data

Post-training:
shape behavior toward useful, controllable, preferred outputs
```

后训练并不是简单“微调一下”。它改变了 AI System 的另一个核心问题：模型能力需要被治理。

如果模型可以回答任何问题，它也可能回答不该回答的问题。如果模型可以生成代码，它也可能生成有风险的代码。如果模型可以调用工具，它就可能触发真实世界动作。能力越强，系统越需要边界。

因此，LLM 时代的生产系统不能只问“模型会不会”，还要问：

- 它是否按照用户意图工作？
- 它是否能在不确定时表达不确定？
- 它是否能遵守业务和安全约束？
- 它是否能稳定输出可解析格式？
- 它是否能在工具调用失败时恢复？
- 它的行为变化是否可以评估、回滚和审计？

这些问题直接连接到后面的 Training System、Evaluation System、Serving System 和 Agent Platform。

## Agent：当模型开始执行动作

LLM 的接口最初像一个更强的文本生成器。用户输入 prompt，模型返回 answer。

但很快，人们发现语言模型不只可以回答问题，也可以作为任务执行的中枢：它可以读取上下文，分解任务，调用搜索、数据库、代码解释器、业务 API，再根据工具返回继续推理和行动。

这就是 Agent 范式的核心变化：模型不再只是生成最终文本，而是参与一个运行时闭环。

```text
Prompt -> Reasoning -> Tool Call -> Observation -> Next Action -> Final Output
```

这个闭环把 AI System 的边界再次扩大。

在普通 LLM Serving 中，主要问题是如何高效、安全地生成 token。在 Agent 系统中，还要处理：

- 工具权限：模型能调用什么工具，能访问什么数据，能执行什么动作？
- 状态管理：当前任务进展、历史对话、用户偏好、工作区文件如何进入上下文？
- Memory：哪些信息应该短期保留，哪些应该长期沉淀，如何避免污染上下文？
- 规划与恢复：任务失败、工具超时、结果冲突时如何继续？
- 可观测性：一次任务经过了哪些推理步骤、工具调用和中间状态？
- 安全边界：如何防止 prompt injection、越权调用、错误操作和不可审计行为？

从系统角度看，Agent 不是一个“更聪明的 prompt”。它是一个由模型、上下文、工具、状态、权限、调度、观测和反馈共同组成的运行时系统。

这也是本书最后一部分要讨论 Agent Platform 的原因。Agent 的难点不只是 prompt 技巧，而是如何把开放式模型能力放进可治理的工程边界。

## 一张瓶颈迁移图

如果把这段历史压缩成一张表，可以看到每一代范式都在解决旧瓶颈，同时制造新瓶颈。

| 阶段 | 核心想法 | 解决的瓶颈 | 新产生的瓶颈 |
| --- | --- | --- | --- |
| 规则系统 | 人显式写出知识和推理规则 | 行为可控、逻辑可追踪 | 覆盖不足、维护困难、泛化弱 |
| 传统机器学习 | 从数据中学习输入到输出的函数 | 不必手写所有规则 | 特征工程、数据分布、评估泛化 |
| 深度学习 | 模型学习多层表示 | 降低手工特征依赖 | 数据规模、算力、优化、可解释性 |
| 序列模型 | 建模时间和上下文依赖 | 处理语言、语音等序列任务 | 长程依赖、并行效率、压缩瓶颈 |
| Transformer | 内容相关的信息聚合和高并行训练 | 更好建模序列关系，适合规模化 | attention 成本、显存、长上下文 |
| LLM | 大规模预训练形成通用条件生成能力 | 多任务通用性、运行时任务定义 | 推理成本、对齐、评估、可靠性 |
| Agent | 模型连接工具、状态和业务流程 | 从回答问题走向执行任务 | 权限、状态、动作安全、观测治理 |

这张表的重点不是分类，而是形成一种系统直觉：

**AI 的每一次进步，都会把原本隐藏在模型之外的问题推入系统内部。**

当模型能力弱时，系统主要围绕规则和特征运转；当模型能力增强，系统开始围绕数据、训练和算力运转；当模型具备通用语言能力，系统开始围绕上下文、推理、评估和治理运转；当模型可以执行动作，系统必须管理权限、状态和责任边界。

这就是 AI System 为什么会越来越重要。

## 从历史看 AI System 的必然性

如果只看算法论文，AI 的历史像是模型结构和训练方法的演进。

如果站在平台工程角度，AI 的历史还有另一条主线：每一代模型能力提升，都会要求新的系统能力承接它。

机器学习需要数据管道、特征平台、训练调度和模型发布。

深度学习需要 GPU 集群、分布式训练、实验管理、checkpoint 管理和大规模数据处理。

Transformer 和 LLM 需要高性能推理引擎、KV Cache 管理、continuous batching、长上下文优化、模型并行、量化、Serving 编排和成本治理。

Agent 需要工具注册、权限控制、memory 管理、workflow 编排、trace、审计、评估和安全防护。

所以 AI System 不是在模型之后额外加的一层工程包装。它是模型能力逐渐生产化、规模化、平台化之后自然出现的系统形态。

这也解释了为什么做 AI Infra 不能只学习某个框架的用法。vLLM、SGLang、TensorRT-LLM、KServe、Kubeflow、Dynamo、MLflow、MCP 等工具，都是某个历史瓶颈在特定阶段的工程回答。理解历史，就是理解这些工具为什么会出现。

## 几条容易误读的历史线索

第一种误读，是把 AI 历史理解成“旧方法被新方法彻底淘汰”。

现实不是这样。规则系统仍然存在，尤其在权限、安全、合规、流程控制和业务约束中非常重要。传统机器学习仍然在推荐、风控、广告、时间序列等场景中大量使用。深度学习、LLM 和 Agent 并不是替代一切，而是在更复杂、更开放的问题上扩展了能力边界。

第二种误读，是把规模看成唯一原因。

规模非常重要。数据、参数和算力的扩大，是 LLM 能力跃迁的重要条件。但规模不是唯一解释。架构、训练目标、数据质量、tokenizer、优化方法、后训练、推理时上下文、工具环境和评估方式，都会影响系统能力。只说“模型变大了所以更强”，会遮蔽很多关键设计。

第三种误读，是把 LLM 当成普通模型服务。

普通模型服务常常是 fixed input -> fixed output。LLM 则更像一个运行时解释器：prompt、context、retrieval、tool result、sampling 参数都会影响行为。它的系统边界天然更宽。

第四种误读，是把 Agent 当成单模型智能。

Agent 的能力不只来自模型参数，也来自工具、上下文、记忆、工作流、环境反馈和系统约束。很多 Agent 问题不是“模型够不够聪明”，而是“运行时系统是否可控、可观测、可恢复”。

第五种误读，是把一次历史突破归因于单个算法。数据规模、硬件、编译器、分布式训练、开源生态和产品形态会共同决定一个方法能否跨过工程可行性门槛。把 Transformer 或 LLM 解释成某篇论文之后自然发生的一切，会遮蔽 AI System 在其中承担的放大作用。

## 对平台工程师的实践含义

理解 AI 历史，对平台工程师的直接价值，是帮助我们判断一个问题属于哪一代瓶颈。

如果线上质量不稳定，不要马上认为需要更大模型。问题可能来自数据分布、prompt 设计、retrieval 召回、上下文截断、sampling 参数、后训练目标或评估口径。

如果 GPU 利用率低，不要只看服务 QPS。LLM 推理的负载单位不是普通请求，而是 prompt token、generated token、上下文长度、KV Cache 占用和 batch 形态。

如果业务要求模型调用内部系统，不要只加 tool calling。要同时设计权限、审计、失败恢复、状态管理和 trace。

如果团队要引入一个新框架，不要先问它的 YAML 怎么写，而要问它回应的是哪一类历史瓶颈：训练规模？显存管理？推理调度？Serving 生命周期？工具连接？多租户治理？

这种判断力，正是本书要建立的核心能力。

## 自检问题

1. 为什么说 AI 的历史不是线性技术堆叠，而是瓶颈迁移？
2. 规则系统为什么在可控性上有优势，却难以覆盖复杂真实世界？
3. 机器学习把系统核心资产从代码扩展到了什么？这对工程流程有什么影响？
4. 深度学习为什么可以减少手工特征工程？它又引入了哪些新的系统瓶颈？
5. RNN 类序列模型为什么在长程依赖和并行训练上遇到限制？
6. Transformer 的 Self Attention 为什么更适合大规模并行训练？
7. LLM 为什么把任务定义从训练期部分移动到了运行期？
8. 为什么后训练和对齐不是可选装饰，而是 LLM 产品化的关键环节？
9. Agent 相比普通 LLM Serving，新增了哪些系统问题？
10. 当你看到一个新的 AI Infra 框架时，如何判断它属于哪一代瓶颈的工程解法？

## 小结

AI 的发展史，不是“模型越来越大、效果越来越好”这么简单。

更准确地说，AI 一直在把人类显式设计的部分逐步交给数据、模型和运行时系统：先是不再手写全部规则，再是不再手写全部特征，然后让模型通过大规模预训练形成通用能力，最后让模型通过上下文和工具参与任务执行。

每一步都扩大了能力边界，也扩大了系统边界。

这正是 AI System 成为独立知识体系的原因。现代 AI 工程不只是训练一个模型、部署一个服务、调用一个 API，而是要理解数据如何生产能力，算力如何支撑能力，推理如何交付能力，评估如何约束能力，Agent 如何把能力连接到真实行动。

理解这条历史线，后面的知识树才不会变成框架清单，而会变成一组可以长期复用的系统判断。
