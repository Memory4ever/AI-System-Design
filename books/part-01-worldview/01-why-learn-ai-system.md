# 第1章 为什么学习 AI System

**Knowledge Tree:** Part I 世界观：AI 为什么会发展成今天这样
**Status:** Draft

**Roadmap Intent:** AI System 不是模型部署，而是让模型稳定、高效、可控、可迭代地产生业务价值的一组系统。

## 本章要回答的问题

一个已经训练好的大模型，为什么不能像传统后端服务一样“部署起来就结束”？为什么 AI 工程的核心不是调用一个模型 API，而是围绕数据、训练、模型、推理、Serving、调度、观测、评估和反馈构建一套系统？

这是整本书的入口问题。

如果这个问题没有想清楚，后面的 vLLM、KServe、Kubeflow、SGLang、Dynamo、LoRA、RLHF、Agent 都会变成一组分散的技术名词。每个名词都能单独学习，但很难形成稳定判断。你可能知道某个框架怎么用，却不知道它到底解决了 AI System 中的哪一类长期问题。

本章先建立一个基本判断：**AI System 不是“模型 + API”，而是让模型能力在真实生产约束下持续产生价值的系统工程。**

模型是核心资产，但模型本身不是完整系统。真正困难的地方在于，模型能力如何产生、如何交付、如何运行、如何被观测、如何被评估，以及如何在业务反馈中持续演化。

## 从一个看似简单的方案开始

假设你所在的平台团队拿到了一个效果不错的 LLM checkpoint。最直接的方案很自然：

把模型放到一台 GPU 机器上，启动推理服务，暴露一个 HTTP API，然后让业务方调用。

第一天，这个方案通常是成立的。少量请求可以返回结果，demo 能跑通，业务方也能看到模型确实具备某种能力。这个阶段很容易产生一个错觉：AI 平台的核心工作就是把模型服务化。

但只要这个 demo 开始接近真实业务环境，问题就会一层一层浮出来。

首先出现的是性能问题。LLM 请求不是普通函数调用。一次请求通常包含 prefill 和 decode 两个阶段，decode 还要逐 token 生成。短 prompt、长 prompt、短输出、长输出，对 GPU 的压力完全不同。传统 Web 服务里，“请求数”常常是一个重要负载指标；但在 LLM Serving 中，只看请求数是不够的。token 数、上下文长度、KV Cache 占用、batching 策略、显存碎片和 decode 速率都会影响吞吐和延迟。

接着是资源问题。GPU 不是普通 CPU 资源。它昂贵、稀缺，而且不同 AI workload 对显存、算力和通信的需求差异很大。训练任务希望长时间稳定占用资源，在线推理任务对 latency 敏感，离线批量推理希望最大化 throughput。只要平台里出现多个团队、多个模型、多个任务优先级，调度就不再是“找一张空卡”这么简单。

然后是质量问题。模型服务即使没有返回 500，也不代表它回答得好。模型输出可能不准确、不稳定、不符合业务策略，或者在某类输入分布下明显退化。传统服务监控关注 QPS、latency、error rate；AI 服务还要关注 answer quality、hallucination、safety、instruction following、retrieval quality 和用户反馈。

最后是演化问题。传统服务上线后主要通过代码迭代；AI 系统的迭代路径要复杂得多。你可能会换模型、调 prompt、改 tokenizer、扩上下文、接 RAG、做 SFT、加 LoRA、调整 sampling、优化推理引擎、修改调度策略、增加评估集，或者引入 tool calling。每一种变化都可能同时影响质量、成本、延迟和稳定性。

所以，“把模型部署起来”只是起点。它证明模型可以被调用，但还没有证明模型能力可以被生产化、平台化和持续治理。

## 传统后端类比为什么会失效

传统软件系统的核心资产通常是代码。代码经过编译、测试、发布之后，只要输入、依赖和运行环境稳定，行为大体可预测。系统复杂性主要来自流量、状态、并发、故障、网络、存储和组织协作。

AI 系统不是这样。

模型不是手写规则，而是从数据中训练出来的参数化函数。它的行为不只由代码决定，还由训练数据、数据分布、模型结构、训练过程、checkpoint、prompt、上下文、解码策略、推理引擎、GPU 资源、线上反馈和评估体系共同决定。

这意味着，当一个团队说“我们已经有模型了”，它实际上只是拥有了一个可能有用的能力载体，还没有拥有可生产化的 AI System。生产环境真正关心的是另一组问题：

- 数据如何进入训练过程，并影响模型能力？
- 模型版本如何管理、评估、回滚和审计？
- 推理如何兼顾 latency、throughput 和 cost？
- GPU 如何被多个团队、多个模型、多个任务共享？
- 服务如何处理突发流量、长上下文、流式输出和失败重试？
- 输出质量如何被持续评估，而不是只靠人工感受？
- 当业务反馈变化时，系统如何支持再训练、微调、RAG 或 prompt 调整？
- 当模型可以调用工具和执行任务时，权限、状态、安全和可观测性如何设计？

这些问题都不是单个模型文件能回答的。它们共同构成 AI System 的问题空间。

## AI System 的复杂性从哪里来

AI System 的复杂性不是某个框架带来的，而是 AI 范式不断演化之后自然出现的。

早期规则系统的核心问题是知识如何表达。工程师把规则写进程序，系统行为可解释，但很难覆盖复杂真实世界。机器学习把规则从手写变成从数据中学习，问题从“如何写规则”变成“如何准备特征、训练模型、评估效果”。这一阶段的系统重点是数据处理、特征工程、离线训练和模型上线。

深度学习进一步减少了手工特征工程，模型可以从原始数据中学习表示。但代价是训练过程更依赖数据规模、算力、优化算法和硬件效率。GPU、分布式训练、checkpoint、实验管理开始成为系统核心。

Transformer 和 LLM 之后，系统边界继续扩大。模型不再只是对固定输入做分类或回归，而是通过 prompt 和 context 在运行时表现出不同能力。推理也不再是一次前向计算，而是持续生成 token 的过程。KV Cache、continuous batching、paged memory、speculative decoding、PD disaggregation 等问题进入基础设施层。

Agent 进一步改变边界。模型可以读取上下文、调用工具、访问外部系统、维护 memory、执行 workflow。此时 AI System 不只是训练和推理平台，还要管理权限、状态、动作、反馈、审计和安全边界。

这条演化路线说明，AI System 不是某个工具栈的拼装结果，而是模型能力不断接近生产环境之后，系统复杂性被逐步展开的结果。

## 从第一性原理看 AI System

从第一性原理看，AI System 解决的是一个约束优化问题：如何在真实资源、真实业务和真实风险下，把不完全确定的模型能力转化为持续可用的系统能力。

这里有三个基本事实。

第一，模型能力来自数据和优化过程，而不是来自手写逻辑。模型行为是训练数据分布、模型架构、目标函数和优化过程共同塑造的结果。因此，模型上线后的问题往往不能只在 Serving 层解决。有些问题需要改数据，有些需要改训练，有些需要改评估，有些只需要改 prompt 或 retrieval。AI System 必须能连接这些层次。

第二，模型运行依赖昂贵且受限的计算资源。大模型推理和训练都受到 GPU compute、GPU memory、interconnect、storage throughput 和调度策略约束。系统设计不能只追求功能可用，还要考虑资源效率。一个质量不错但成本不可控、延迟不可控、无法扩容的模型服务，在生产上仍然不可用。

第三，模型输出具有概率性和上下文依赖。即使同一个模型，在不同 prompt、context、sampling 参数和工具环境下，也可能表现出不同结果。因此 AI System 需要评估、观测、反馈和治理机制，把“模型可能有用”变成“系统可以被信任地使用”。

学习 AI System 的核心，不是先记住某个框架的 API，而是建立一组跨层判断：

- 当前问题发生在数据层、模型层、训练层、推理层、Serving 层，还是平台治理层？
- 这个问题应该通过模型能力提升解决，还是通过系统约束解决？
- 当前瓶颈是质量、延迟、吞吐、成本、稳定性、安全，还是组织协作？
- 如果未来模型更大、上下文更长、工具更多、用户更多，现有设计是否仍然成立？

## 一个更工程化的判断框架

生产级 AI System 不是单目标问题，而是多目标约束优化。可以把一个 AI 服务的价值粗略理解为：

```text
System Value = Business Utility(Quality, Coverage, Freshness)
             - Cost(Training, Inference, Storage, Operation)
             - Risk(Hallucination, Safety, Privacy, Compliance)
```

这里的 `Quality` 不是单一准确率。对于 LLM 应用，它可能包含事实正确性、指令遵循、格式稳定性、推理能力、召回能力、安全性和用户满意度。`Cost` 也不只是机器费用，还包括 GPU 使用、工程维护、数据治理、评估构建、排障成本和平台复杂度。`Risk` 则来自模型不确定性、数据泄露、工具误调用、权限边界和不可解释行为。

如果站在推理系统角度，还可以把问题表达成：

```text
maximize   throughput
subject to latency <= SLO
           GPU memory <= capacity
           cost per request <= budget
           quality >= acceptable threshold
           isolation and safety constraints are satisfied
```

这个表达不追求数学严密，而是帮助我们形成正确的系统直觉：AI System 的设计从来不是单纯追求“模型效果最好”或“吞吐最高”。真正的工程目标是在多个约束之间找到可持续的平衡点。

这也是为什么本书后面会同时讨论模型原理、训练系统、推理系统、Serving、调度、可观测性和 Agent。它们不是并列的知识点，而是同一个多目标系统的不同约束面。

## AI System 至少包含五条链路

从工程实现角度看，一个 AI System 至少需要覆盖五条链路。

第一条是能力生产链路：Data → Training → Checkpoint → Evaluation。数据经过清洗、标注、去重、配比和训练，形成模型权重。checkpoint 需要被保存、比较、评估和版本化。没有这条链路，模型能力无法被系统性生产和复现。

第二条是能力交付链路：Model Registry → Runtime → Serving → Gateway → Client。模型文件需要进入可运行环境，由推理引擎加载，通过 Serving 层对外暴露，再经过网关完成认证、限流、路由和协议适配。没有这条链路，模型无法稳定服务业务。

第三条是资源管理链路：GPU Pool → Scheduler → Workload → Utilization → Cost。训练、微调、离线推理、在线推理都会竞争 GPU。平台需要处理配额、优先级、隔离、弹性、利用率和成本归因。没有这条链路，AI 能力无法规模化。

第四条是质量反馈链路：Request → Output → Evaluation → Feedback → Iteration。AI 系统不能只观察服务是否报错，还要观察输出是否有用。评估集、线上反馈、人工审核、A/B 实验和 trace 都是质量闭环的一部分。

第五条是 Agent 运行链路：Prompt → Context → Memory → Tool → Action → Observation。模型一旦开始调用工具和执行任务，系统就必须管理状态、权限、动作边界、失败恢复和审计。这使 AI System 从模型服务扩展为运行时编排系统。

这些链路共同说明：AI System 的工程形态更像一个平台，而不是一个单体服务。

## 设计时真正困难的不是“有没有方案”

AI System 的难点通常不在于是否存在某个技术方案，而在于不同目标之间互相拉扯。

质量和成本之间存在张力。更大的模型、更长的上下文、更复杂的 RAG、更严格的评估通常可以提高质量，但会增加 GPU、存储、延迟和工程复杂度。便宜的方案可能足够覆盖部分场景，但在复杂任务上不稳定。

延迟和吞吐之间存在张力。在线交互希望低 latency，而 GPU 利用率往往依赖 batching。batch 太小，资源浪费；batch 太大，用户等待。推理系统设计必须在用户体验和资源效率之间找平衡。

通用性和可控性之间存在张力。一个通用模型可以覆盖更多任务，但输出更难约束；一个强约束 workflow 更稳定，但灵活性下降。Agent 系统尤其需要处理这个矛盾。

平台抽象和性能优化之间存在张力。平台希望提供统一接口、统一治理和统一生命周期；底层推理引擎又高度依赖模型结构、GPU 类型、并行策略、KV Cache 管理和调度细节。抽象太薄，业务使用成本高；抽象太厚，性能和能力可能被隐藏。

短期交付和长期知识树之间也有张力。工程团队常常需要快速接入一个框架解决当前问题，但如果没有全局知识树，框架经验会变成碎片。学习 AI System 的目的，就是把这些局部技术放回长期问题中理解。

## 几条容易走偏的路径

第一种误区是框架优先。比如先学 vLLM、KServe、Kubeflow、SGLang、Dynamo，再试图把它们拼成知识体系。这条路径短期有效，因为框架有文档、有示例、有 API。但它容易造成一种错觉：掌握框架等于掌握系统。问题在于框架会变化，而底层问题更稳定。只有理解了推理调度、显存管理、模型 Serving、平台治理，才能判断一个框架为什么这样设计，以及什么时候不适合使用。

第二种误区是模型优先。只关注模型效果、参数规模、benchmark 和论文结论，却忽略数据、推理、成本和上线治理。这会导致模型在实验环境里看起来很好，但进入生产后无法满足 SLO、成本预算、可观测性和安全要求。

第三种误区是后端服务类比。把 LLM API 当成普通 HTTP 服务，把 token 生成当成普通函数调用，把 GPU 当成普通计算资源。这种类比会遮蔽 LLM 推理的关键差异：自回归生成、KV Cache、显存约束、batching 策略和输出不确定性。

第四种误区是一次性交付。认为模型上线就是结束。实际上，AI System 的价值来自持续迭代：数据更新、模型升级、评估改进、推理优化、用户反馈、成本治理和安全策略都会持续变化。

这些路径不是完全错误，它们在某些阶段都有用。但如果把它们当成完整方法，就会进入死胡同。

## 对平台工程师的实践含义

对有基础设施背景的工程师来说，学习 AI System 可以从几个实践判断开始。

遇到一个新框架时，先不要问“API 怎么用”，而要问它解决哪个长期问题。vLLM 这类推理引擎的核心不是暴露 OpenAI-compatible API，而是解决 LLM Serving 中的吞吐、显存和调度问题。KServe 这类 Serving 平台的核心不是一个 YAML，而是模型服务生命周期、Runtime 抽象和流量治理。Kubeflow 的核心不是 UI，而是在 Kubernetes 上组织训练、Pipeline 和平台能力。

设计一个 AI 平台功能时，要先确定它属于哪条链路。是能力生产、能力交付、资源管理、质量反馈，还是 Agent 编排？不同链路的指标、故障模式和抽象边界都不同。

排查一个线上 AI 问题时，也要避免只看服务日志。一次输出质量下降，可能来自模型版本变化、prompt 变化、retrieval 质量下降、输入分布漂移、sampling 参数变化、上下文截断、工具失败或评估口径变化。AI System 的 observability 必须能跨越模型、数据、请求、上下文和资源。

做技术选型时，还要把未来约束纳入判断。今天只有一个模型、一个团队、少量请求时，简单部署足够；当模型数量增加、请求量增长、团队共享 GPU、需要审计和成本归因时，平台抽象就变得必要。反过来，在 PoC 阶段过早引入过重平台，也会拖慢学习速度。

## 本章在全书中的位置

本章是整棵知识树的入口。它不深入某个具体机制，而是定义为什么需要这棵树。

后续章节可以看作对本章问题的逐层展开：

```text
为什么学习 AI System
→ AI 的发展历史
→ AI System 全局知识树
→ 模型为什么能够学习
→ Transformer / LLM 内部机制
→ Training System 如何生产模型能力
→ Inference System 如何高效运行模型
→ AI Infrastructure 如何平台化能力
→ Agent 如何把模型连接到任务执行
```

如果没有本章的问题意识，后面的内容容易变成框架清单。如果理解了本章，后面的每个技术点都可以被放回一个统一问题中：它到底解决 AI System 的哪个约束？

## 自检问题

1. 为什么说 AI System 不是简单的模型部署？
2. 传统后端服务和 LLM Serving 在负载特征上有什么关键差异？
3. 为什么 GPU 调度会成为 AI 平台的核心问题？
4. 模型输出质量为什么不能只靠 error rate 监控？
5. 如果一个模型 demo 效果很好，进入生产前还需要补齐哪些系统能力？
6. 为什么学习 vLLM、KServe、Kubeflow 之前，需要先建立 AI System 知识树？
7. RAG、Fine-tuning、Prompt Engineering 分别更像在改变系统的哪一层？
8. Agent 系统为什么会把 AI System 的边界从 Serving 扩展到运行时编排？

## 小结

学习 AI System 的第一步，是放弃“我只需要会部署模型”的想法。

对于有分布式系统和平台工程背景的工程师来说，AI System 并不是完全陌生的领域。它仍然关心资源、调度、服务、观测、稳定性、成本和治理。但它引入了一个新的核心资产：模型能力。这个资产来自数据和训练，运行时依赖上下文和推理系统，输出又具有概率性和不确定性。

所以，AI System 的学习目标不是追逐每一个新名词，而是建立判断力：看到一个新框架时，能判断它解决什么问题；看到一个线上现象时，能定位它属于哪一层；设计一个平台能力时，能理解它和整棵知识树的关系。

从这一章开始，我们要建立的不是工具记忆，而是一张稳定的系统地图。
